Airline Neutral

The Project

This project was built for a renowned airlines and its newly acquired airlines. The aim of the project was to make the existing mobile app airline neutral - a design that would be based on same structure for any number of sub-products and has to be consistent across all the platforms.

The app was to be redesigned for all the platforms, viz., mobile web, iOS and Android.

My Role

Analysis

Analysis of existing product with the business analyst and product owner

Strategy

Devised a strategy to build the UI design for an airline neutral product

UI Design

Lead the implementation of UI design across iOS, Android and Mobile Web with developers

Leading UI development

Lead and implemented SASS code to build a scalable solution across platforms

The Approach

I partnered with the product owner and business analyst to analyse the plan and we decided to carry out the project in 4 phases:

1. Strategy and Analysis

The project aimed at having its first release after 4 months. So, we needed a plan that would help us to carry out the 'Airline neutral' project effectively.

In the beginning, we accessed our existing designs and code base. After meetings with tech leads, we came to know that code base was tightly coupled to the parent airlines. It was built with the aim of only one airlines, hence code was not at all reusable.

So, our first step was to direct the teams, viz., mobile web, Android and iOS, to decouple and refactor the code to make it scalable and reusable.

Next, we got some key screens (2-3 in number) of the new airlines UI from the design agency. The expectation from the business was to study these key screens and repaint the app for the new airlines. The problem was not just repainting, the problem was to build an airline-neutral solution which could scale from the parent app and can accomodate any number of airlines in the future.

2. Build repaint directives

We had received 2-3 key screens from the design agency consisting of the original app in new brand skin. Our task was to create directives based on these designs to create logical transformations that could easily apply throughout the application for all platforms, viz., mobile web, Android and iOS.

I partnered with the product owner to analyse those key screens and extracted out some rules. We then went through all the screens of the parent mobile web app and analysed how the rules would fit into them. After this rigorous exercise, we were able to create a map representing logical transformation of foundational styles and components. Example:

  • Parent Airlines New Airlines Scenario
  • Brand Blue Light Grey 1 Used in backgrounds
  • Brand Red Brand Gradient Primary buttons

3. Repaint mobile web

Since I was also the UI developer on the project, I was responsible to refactor and decouple the SASS codebase as well. In order to understand the refactoring process in a better way, let’s take a car analogy. Let’s take model name called 'Alpha' for the analogy.

Inconsistent car components

Imagine a scenario in which each and every car of model 'Alpha' is different. Every car of the model has its own unique steering wheel, motor, etc. These inconsistencies may lead to unnecessary confusion and delays. Also, the maintenance would cost a lot.

Similary, our product had a number of inconsistencies. We jotted down all of them. Few of them were:

  • inconsistent use of buttons for primary and secondary action,
  • different colour hex values for same colour,
  • different colours for links,
  • inconsistent design for forms,
  • inconsistent alignment issues
Fixing inconsistencies

Next step was to fix these inconsistencies.

We worked to remove redundant and inconsistent UI components. We ensured that the styles are clean and unambiguous across the mobile web.

For example, we had 6 different shades of primary blue. So, we discussed the brand guidelines with the marketing team and business and removed the ambiguity in the brand blue. If the inconsistencies are removed, every component or style becomes unique and independent entity which can be easily replaced or used polymorphically.

This is similar to the fact that, if the car mechanic knows the model of the car and its parts well, he/she can confidently deal with the car, fix any issues and can even replace or optimise something to reduce the cost.

Extracting out components

The car could now be disintegrated into its sub components - brakes, window, steering wheel, etc. Each component is unique and independent of each other. Also, each component is made up of sub-components which are clearly defined.

After extraction of all the car components, a skeleton or the wireframe of the car is left.

Our task was to breakdown the product into SASS components. Since we had a huge amount of legacy code, we carried a hybrid approach of atomic design and component-based depending upon the need of our product.

Then, we carried refactoring of the SASS code to extract out atoms, molecules and various UI components.

We extracted out the structure of the product, i.e, layouts, templates, etc. - these will remain same for both the airlines. Now our task of making components airline-neutral came into picture.

Building a new model

Next, refactored and reusable SASS components were repainted with the new visual guidelines extracted in the previous phase. Since the structure of the car remained the same, the newly styled car components can now be used to build up a new model of the same car.

Similarly, the newly painted SASS components could now be put together into the same layout to build a new application. In technical terms, the base code now behaved polymorphically according to the airlines.

Here comes the new model!

After putting together the components, we got our new product.

Feedback

I shared the newly painted and refactored app with the Android and iOS team. I translated my approach to the developers and handed them guidelines to carry out the repainting.

Translation of the approach to the iOS and Android teams

4. Rebrand Android and iOS

After receiving feedback from iOS and Android developers, my aim was to implement these guidelines across both these platforms.

Therefore, I paired with the Android and iOS developers to implement the repainting correctly. We had to ensure that the repaint occurs uniformly across all platforms. This helped me to refine the guidelines further and cater to OS-specific needs.

Visual Regression Testing

I partnered with the QA to build up a ruby gem 'Nakal' to carry out automated visual regression testing of the app. This helped us to carry out visual validations for our product while development.
