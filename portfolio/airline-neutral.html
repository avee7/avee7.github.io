Airline Neutral

UI design Strategy and Planning UI Development

This project was built for an eminent US airlines and its newly acquired airlines (name is confidential).

The newly acquired airlines product was redesigned from the parent airlines product for mobile web, iOS and Android.

My Role

Analysis

Analysed the parent product with the business analyst and product owner to plan the course of action.

Strategy

Devised a strategy to build the UI design for an airline neutral product

UI Design

Lead the implementation of UI design across iOS, Android and Mobile Web with developers

UI development

Lead and implemented SASS code to build a scalable solution across platforms

Project

Building an Airline neutral product

The airline neutral project wanted to build a single product which could wear different visual styles depending upon the sub-product.

The aim of the project was:

  • to design a user interface based on the parent airlines - on mobile web, iOS and Android app.
  • to build an airline-neutral solution which is based on the parent airlines and can accomodate any number of airlines in future.

Approach and Planning

We built an approach that would allow multidisciplinary teams (Mobile web, iOS and Android) to progress in their own direction with a set of directives.

Our aim was to enable the development teams to understand the logic behind the UI design and carry out development with confidence.

In partnership with the product owner and business analyst, we decided to carry out the project in 4 phases.

Four steps of our planning

The Process

Problem Analysis

In the analysis phase, we accessed our existing designs and code base. We directed the dev teams (mobile web, iOS and Android) to inspect their code base and start in the direction of making it modular and reusable.

We also analysed the parent product to define its visual styles and language.

Build Directives

Next, we designed representative screens from all platforms to define a visual design for the new product based on the brand guidelines.

Logical transformations were created between the parent product and the sub-product based on brand colours, components and patterns keeping the user journeys same. Here's an example of a logical transformation comprising of foundational styles and components of the design.

Identify and Fix Inconsistency

Wearing the UI developer hat, I started refactoring the mobile web code to make it modular and reusable.

Inconsistent styles or components were identified and removed. Redundant styles were converged which further refined our directives.

For example, we fixed 6 different shades of primary blue to one.

Extract Structure

The task was to separate structure and skin of the product.

We extracted out layouts, templates, etc. - which remained same for both the airlines.

Car analogy: Extracting structure from the product

Redesign Components

In mobile web, we transformed the foundational styles and redesigned SASS components. Since we had a huge amount of legacy code, we carried a hybrid approach of atomic design and component-based design.

This further refined our rebrand directives.

Car analogy: Redesign car components

New model for Mobile Web

The newly rebranded components were put together inside the same structure to build a new model.

In technical terms, we built a source code for mobile web which could behave polymorphically.

Multiple products using structure of one product and multiple brands

Enable iOS and Android UI team

After multiple refinements and feedback, we shared the directives with the iOS and Android team.

I partnered with the iOS and Android teams to enable them carry out the product redesign with the set of directives. It was also ensured that our guidelines are consistent for all the platforms.

Partnership with the QA

In order to reduce the time to QA the visual styles, I partnered with the QA team to build up a ruby gem 'Nakal' to carry out automated visual regression testing. This helped us to carry out visual validations for our product while development.
